# OpenSteamTool GUI 实施文档

> 本文档由设计访谈（2026-05-17）产出，记录 GUI 前端的全部设计决策与实施规范。

## 1. 项目定位

**OpenSteamTool GUI** 是 OpenSteamTool C++ DLL 的配套图形化管理工具，定位为：

| 模块 | 功能 |
|------|------|
| B — 解锁库管理器 | 按游戏维度管理 appid / DLC / depot key / access token / manifest binding / AppTicket / ETicket / Stat steamid |
| C — 安装器 + 日志查看器 | DLL 安装/卸载（拷贝 dwmapi.dll / xinput1_4.dll / OpenSteamTool.dll 到 Steam 根目录）+ 解析 `<Steam>/opensteamtool/*.log` 12 个分类日志 |
| D — 数据获取前端 | 对接 Steam Web API / CaiGames / SteamCMD / GitHub 清单库 / Sudama 全量库，半自动生成解锁配置 |

GUI 与 DLL 之间**不通过 IPC**，仅通过文件系统协作（写 Lua、读日志）。DLL 已有 FileWatcher 热重载机制。

## 2. 技术栈

| 项 | 选型 | 理由 |

## 3. 数据架构

### 3.1 真相源策略

采用「**DB 主导 + 隔离生成目录**」混合模式：

- GUI 内部 **SQLite**（WAL 模式）存游戏库全部元信息
- 生成目录：`<Steam>/config/lua/managed/`（DLL 默认能加载到）
- 每个游戏一个 `.lua` 文件：`app_<main_appid>.lua`
- 旧 Lua 文件在 GUI 中显示为「外部条目」只读视图，不主动修改

### 3.2 DB Schema

```sql
-- 游戏/DLC 统一表
CREATE TABLE games (
    appid        INTEGER PRIMARY KEY,
    name         TEXT NOT NULL DEFAULT '',
    type         TEXT NOT NULL DEFAULT 'game',     -- game/dlc/tool/demo
    parent_appid INTEGER REFERENCES games(appid),
    header_image TEXT,                              -- 本地缓存路径或远程 URL
    source       TEXT,                              -- API/CaiGames/SteamCMD/GitHub/Sudama/Manual/Imported
    fetched_at   TEXT,                              -- ISO 8601
    enabled      INTEGER NOT NULL DEFAULT 1,
    incomplete   INTEGER NOT NULL DEFAULT 0,        -- 字段缺失标记
    managed      INTEGER NOT NULL DEFAULT 1,        -- 1=GUI 托管，0=外部 Lua 只读
    note         TEXT,
    created_at   TEXT NOT NULL,
    updated_at   TEXT NOT NULL
);

CREATE INDEX idx_games_parent ON games(parent_appid);
CREATE INDEX idx_games_type   ON games(type);

-- depot（depot_id 全局唯一）
CREATE TABLE depots (
    depot_id        INTEGER PRIMARY KEY,
    owner_appid     INTEGER NOT NULL REFERENCES games(appid) ON DELETE CASCADE,
    decryption_key  TEXT,                            -- 64 字符 hex 或 NULL
    manifest_gid    TEXT,                            -- decimal string
    key_source      TEXT,                            -- 来源标记（GitHub repo / Sudama / Manual）
    updated_at      TEXT NOT NULL
);

CREATE INDEX idx_depots_owner ON depots(owner_appid);

-- 单值附加配置（按 appid 一对一）
CREATE TABLE access_tokens  (appid INTEGER PRIMARY KEY REFERENCES games(appid) ON DELETE CASCADE, token TEXT NOT NULL);
CREATE TABLE app_tickets    (appid INTEGER PRIMARY KEY REFERENCES games(appid) ON DELETE CASCADE, ticket_hex TEXT NOT NULL, written_to_registry_at TEXT);
CREATE TABLE e_tickets      (appid INTEGER PRIMARY KEY REFERENCES games(appid) ON DELETE CASCADE, ticket_hex TEXT NOT NULL, written_to_registry_at TEXT);
CREATE TABLE stat_overrides (appid INTEGER PRIMARY KEY REFERENCES games(appid) ON DELETE CASCADE, steam_id TEXT NOT NULL);

-- 注册表备份（写 AppTicket/ETicket 前的兜底）
CREATE TABLE registry_backups (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    appid        INTEGER NOT NULL,
    value_name   TEXT NOT NULL,                     -- 'AppTicket' or 'ETicket'
    value_hex    TEXT,                               -- 备份的原值（NULL 表示原本不存在）
    backed_up_at TEXT NOT NULL
);

-- GUI 自身设置
CREATE TABLE settings (key TEXT PRIMARY KEY, value TEXT NOT NULL);
```

### 3.3 启用/禁用语义

- **双层粒度**：游戏级 + DLC 级
  - 游戏整体禁用 → 不生成 `.lua` 文件
  - 游戏启用但 DLC X 禁用 → 文件中跳过该 DLC 的所有调用
- **AppTicket / ETicket 独立**：写入注册表操作与 Lua 生成解耦，有独立「写入/清除」按钮
- **禁用 ≠ 删除**：禁用保留 DB 记录，仅不生成 Lua；删除才移除记录 + 文件

### 3.4 Lua 文件生成

- 粒度：一个游戏一个文件 `app_<main_appid>.lua`，DLC 写在主游戏文件内
- 文件头注释：`-- Generated by OpenSteamTool GUI. DO NOT EDIT MANUALLY.`
- 内容顺序固定（便于 diff）：主游戏 addappid → addtoken → setManifestid → setAppTicket → setETicket → setStat → 每个 DLC 同样顺序
- 写盘原子性：写 `.tmp` → `os.replace()` 原子重命名

## 4. 运行模式

- **默认**：独立工具，双击打开 → 编辑 → 保存 → 关闭即退出
- **可选**：设置勾选「关闭时最小化到托盘」→ 托盘菜单（显示主窗口 / 日志查看器 / 退出）
- **不做**：Steam 启动器外壳、开机自启、单实例锁

## 5. Steam 路径定位

按优先级尝试：
1. 注册表 `HKCU\Software\Valve\Steam` → `SteamPath`
2. 注册表 `HKLM\Software\WOW6432Node\Valve\Steam\InstallPath`
3. 默认 `C:\Program Files (x86)\Steam`
4. 用户手动选择（必须含 `steam.exe`）

路径持久化到 GUI 设置，每次启动校验一次。不做多 Steam 实例支持。

### 6.5 批量操作

- 主题：跟随系统 / 强制亮色 / 强制暗色（三选一）

### 6.7 视觉风格设计语言（金属箔 / 彩箔）

#### 6.7.1 设计定位

参照「金属箔/彩箔」风格，把 GUI 渲染成精致彩箔卡：闪耀箔面承载视觉重量，压纹与高亮折射表达高级感与收藏感，深色留白区域承载文字保持可读。整体氛围：**奢华、收藏、闪亮**，但不能干扰阅读。

适用场景类比：高级包装 / 票券 / 潮流品牌周边。这套语言对应到 OpenSteamTool 的内容（游戏库、解锁配置）天然契合「收藏卡」隐喻——每张游戏卡片就是一张彩箔藏品。

#### 6.7.2 色彩体系

**底色（深色基底，承载文字）**：

| token | 暗色 | 亮色 | 用途 |
|-------|------|------|------|
| `--bg-base`     | `#0B0B10` | `#F5F3EE` | 窗口背景 |
| `--bg-surface`  | `#15151D` | `#FFFFFF` | 卡片/面板表面 |
| `--bg-elevated` | `#1E1E2A` | `#FAF7F0` | 悬浮层（弹窗、popover） |
| `--text-primary`   | `#F2EDE0` | `#1A1A22` | 主文本 |
| `--text-secondary` | `#A8A39A` | `#5C5A55` | 次文本 |

**金属箔渐变（核心装饰）**：四套预设，按区域语义切换。

| token | 渐变（线性 135°） | 语义 |
|-------|------|------|
| `--foil-prism`  | `#FFB7E5 → #B7D8FF → #C0FFD8 → #FFE7B0 → #E8B7FF` | 主品牌色（封面光带、Logo 边） |
| `--foil-gold`   | `#F2D58A → #FFF1B8 → #C8A24A → #FFEFB8 → #B07F2A` | 已启用 / 配置完整状态 |
| `--foil-silver` | `#D4D8E0 → #FFFFFF → #9FA4B0 → #ECEEF2 → #7A7E88` | 默认装饰、按钮金属描边 |
| `--foil-rose`   | `#FFC2D8 → #FFE4EC → #C97A93 → #FFD4E0 → #8C3F5A` | 警告 / 缺失配置 chip |

**功能色（不参与彩箔，确保对比度）**：

- `--accent`：`#7B7BFF`（暗）/ `#5959E6`（亮），CTA 按钮、链接、焦点环
- `--success`：`#3ECF8E`（仅图标着色，不做大面积填充）
- `--warning`：`#E8B14A`
- `--danger`：`#FF5C7A`

#### 6.7.3 材质与质感

- **镜面高光**：在彩箔渐变上叠加 `radial-gradient(ellipse at 30% 20%, rgba(255,255,255,0.45), transparent 60%)` 模拟反光斑点
- **金属噪点**：4×4 像素自定义噪点纹理（PNG，`opacity: 0.06–0.12`），叠在彩箔层上打散平滑感，避免「塑料假光」
- **压纹线**：1px `rgba(255,255,255,0.08)` 内描边 + 1px `rgba(0,0,0,0.35)` 内阴影，形成微浮雕分割
- **金属描边**：`--foil-silver` 渐变作为 1px conic 边框（用 `QPainter` 自绘，PySide6 QSS 不支持渐变 border）
- **不用雾面/磨砂**：会吃掉金属反光，与定位冲突
- **圆角**：卡片 `12px`、按钮 `8px`、chip `999px`（胶囊）

#### 6.7.4 关键组件视觉规范

**主表格行（游戏卡片）**：
- 高度按行高档位：紧凑 36 / 标准 44 / 舒适 56（比默认高 4-12px 留出箔面呼吸感）
- 启用行：左侧 4px 竖条用 `--foil-gold` 渐变 + 噪点
- 禁用行：左侧 4px 竖条用 `--foil-silver` + 50% 不透明度
- 缺配置行：左侧 4px 竖条用 `--foil-rose`
- 选中行：整行背景 `linear-gradient(90deg, rgba(123,123,255,0.12), transparent 60%)` + 1px 顶/底压纹线
- 封面缩略图：圆角 6px + 1px 银箔描边

**详情面板顶部标头**：
- 96×96 封面外框：3px conic 渐变描边（用 `--foil-prism`）+ 1px 内压纹线
- 名称下方：1px 高的彩箔分割条（`--foil-prism`，宽度跟随名称文字宽度）
- 「保存」按钮：未保存态用 `--foil-gold` 渐变填充 + 镜面高光 + 噪点；已保存态退化为 `--foil-silver` 描边幽灵按钮

**主导航（左侧 4 项）**：
- 选中项：背景 `linear-gradient(180deg, rgba(255,255,255,0.06), rgba(255,255,255,0.02))` + 左侧 3px `--foil-prism` 竖条 + 文字提亮
- 未选中：纯 `--text-secondary`，hover 时背景出现 `rgba(255,255,255,0.04)`

**Chip / 状态标签**：
- 完整配置：金箔填充 + 深色文字
- 缺失配置：玫瑰箔 1px 描边 + 透明填充 + warning 色文字
- 来源标签：银箔描边 + `--text-secondary` 文字

**对话框 / 弹窗**：
- 表面 `--bg-elevated` + 12px 圆角 + 1px 银箔描边
- 顶部 80px 区域用淡彩箔渐变（10% 不透明度）做装饰带
- 关键 CTA 按钮（添加 / 安装 / 删除确认）用对应语义箔面填充

**安装器进度 / 加载状态**：
- 进度条：彩箔渐变扫光（150-220ms 循环平移），不用纯色填充
- 长任务等待：标题旁的小型圆形指示器用 conic 渐变旋转

#### 6.7.5 交互动效

**Hover 光扫（核心交互）**：
- 卡片 / 按钮 hover 时一道宽 30% 的高光从左到右扫过（背景 `linear-gradient(105deg, transparent, rgba(255,255,255,0.18), transparent)` 移位 `-30% → 130%`）
- 时长 220ms，缓动 `cubic-bezier(0.22, 0.61, 0.36, 1)`
- 同时噪点纹理亮度从 0.08 提升到 0.14（`QGraphicsOpacityEffect` 控制）

**Active 下沉**：
- 按钮 active：`translateY(1px)` + 阴影从 `0 4px 12px rgba(0,0,0,0.3)` 收敛到 `0 1px 4px rgba(0,0,0,0.4)`
- 光带在 active 瞬间收缩到中心（光斑收敛动效，180ms）

**焦点环**：
- 2px `--accent` 实色环 + 4px `rgba(123,123,255,0.25)` 外辉光
- 不参与彩箔渲染，纯功能性

**全局动效约束**：
- 默认时长 150-220ms，不做 >300ms 的动画
- 用户可在「设置 → 视觉」勾选「减少动效」（对应 Windows 系统 `prefers-reduced-motion`）→ 关闭所有光扫，hover 退化为简单背景色变化
- 帧率目标：60fps；高 DPI 屏 + 多卡片同时 hover 时动效降级（每帧 >16.6ms 自动关闭噪点动画层）

#### 6.7.6 PySide6 实施层

PySide6 / Qt 不直接支持 CSS 渐变 border、conic-gradient、噪点纹理叠加等高级特性，需要**混合方案**：

**1. QSS（基础控件样式）**：
- 颜色 token 写成 QSS 变量风格（用模板字符串注入到运行时 `QApplication.setStyleSheet`）
- 圆角、字体、间距、纯色背景、简单线性渐变（QSS 自带 `qlineargradient`）走 QSS

**2. 自定义 paintEvent（彩箔效果）**：
- 封装 `FoilWidget` 基类：在 `paintEvent` 中按层绘制（底色 → 渐变 → 噪点 PNG → 高光 → 边框 → 内容）
- 用 `QLinearGradient` / `QConicalGradient` / `QRadialGradient` 组合实现镜面高光
- 噪点 PNG 作为 `QPixmap` 资源 tile（`drawTiledPixmap`），透明度通过 `QPainter::setOpacity` 控制
- 主要派生类：`FoilCard`（详情面板封面框）、`FoilButton`（CTA 按钮）、`FoilChip`（状态标签）、`FoilDivider`（彩箔分割条）

**3. QPropertyAnimation（光扫动效）**：
- 自定义 Q_PROPERTY `shimmerOffset`（float，-0.3 → 1.3），驱动 paintEvent 重绘
- Hover 进入触发动画，离开时反向播放或冻结
- 全局动效开关通过 `QApplication` 单例属性控制

**4. 资源管线**：
- 噪点 PNG：64×64 tile，编入 Qt resource（`.qrc`），`qrc:/foil/noise.png`
- 占位封面 SVG：`qrc:/icons/placeholder.svg`
- 不内嵌字体（使用系统字体）

**5. 主题切换**：
- QSS 模板用 Python f-string 在运行时按当前 token 字典渲染
- 切换主题时重新调用 `setStyleSheet` + `update()` 所有 `FoilWidget` 实例（用 `QApplication.allWidgets()` 遍历）

#### 6.7.7 可访问性与降级

- **对比度**：彩箔区域上**不**直接放正文文本，文字一律落在 `--bg-surface` / `--bg-elevated` 区域上，确保 WCAG AA（>=4.5:1）
- **色盲友好**：状态信息不能仅靠箔色区分，必须叠加图标 / 文字 chip
- **低端机降级**：检测 `QSurfaceFormat` 渲染能力，软件渲染时关闭噪点层与光扫动效（保留渐变与压纹）
- **暗色优先**：金属箔在深色基底上反光最强，亮色主题做了适配但视觉重量低 30%（通过降低渐变饱和度实现）

## 7. C 模块（一）：DLL 安装器

### 7.1 部署文件清单

来自 `src/CMakeLists.txt:74-141`：

| 文件 | 角色 |
|------|------|
| `OpenSteamTool.dll` | 主 hook DLL（静态 MSVC 运行时，无额外依赖） |
| `dwmapi.dll` | loader DLL |
| `xinput1_4.dll` | loader DLL（带 .def） |

全部 x64。

### 7.2 安装来源

按优先级三档：

1. **内置（默认）**：随 GUI 打包的 Release DLL，位于 `<gui_install>/dlls/`，CI 注入
2. **在线更新（推荐）**：从 GitHub Releases 拉取上游最新版（详见 7.7）
3. **本地文件（高级）**：用户「使用本地文件」三选文件对话框（自定义 / Debug 用）

GUI 与 DLL 版本绑定关系记录在 `dlls/manifest.json`：

```json
{
  "version": "1.0.0",
  "files": {
    "OpenSteamTool.dll": "<sha256>",
    "dwmapi.dll": "<sha256>",
    "xinput1_4.dll": "<sha256>"
  },
  "built_at": "2026-05-17T14:23:11Z"
}
```

### 7.3 安装状态四档

通过 SHA256 比较内置 DLL 与 `<Steam>/` 中的同名文件：

- 未安装（三个文件都不存在）
- 已安装且版本匹配（哈希全部一致）
- 已安装但版本不同（至少一个哈希不一致）
- 部分安装（只有部分文件存在）

状态显示在状态栏 + 安装器页醒目位置。

### 7.4 安装前安全检查

1. **Steam 进程检测**：用 `psutil` 检测 `steam.exe`，若运行 → 弹「请关闭 Steam」对话框 + 「关闭 Steam」按钮（`taskkill /IM steam.exe`，二次确认）
2. **路径校验**：目标目录必须含 `steam.exe`
3. **权限检查**：检测当前进程是否管理员（`ctypes.windll.shell32.IsUserAnAdmin()`），不足时弹「以管理员重启」按钮（`ShellExecuteW(None, "runas", sys.executable, ...)`）

### 7.5 备份与回滚

- 同名文件备份到 `<Steam>/opensteamtool-gui/backup/<ISO_timestamp>/<filename>`
- 保留最近 5 次，超出自动清理（按时间戳排序删除最旧）
- 「回滚」按钮：列出所有备份，选中后还原对应文件

### 7.6 卸载流程

1. 删三个 DLL
2. 询问可选清理（三个独立复选框，**默认全不勾**）：
   - 删除日志目录 `<Steam>/opensteamtool/`
   - 删除 GUI 生成目录 `<Steam>/config/lua/managed/`
   - 删除 `<Steam>/opensteamtool.toml`
3. **不删**用户手写的 `<Steam>/config/lua/*.lua`

### 7.7 在线更新检测（OpenSteamTool DLL）

GUI 内置 DLL 是发版时刻的 Release 快照；OpenSteamTool 上游更新频繁（hook 模式可能随 Steam 客户端更新而失效），所以 GUI 必须能在不重新发版的情况下，让用户拉到最新 DLL。

#### 7.7.1 上游 Release 约定

- **仓库**：`OpenSteam001/OpenSteamTool`（用户在第二条需求中给出的 URL 前缀已确认）
- **Release Tag 格式**：`v<MAJOR>.<MINOR>.<PATCH>`（例：`v1.4.3`）
- **资产命名约定**：`OpenSteamTool-<tag>-Release.zip`（例：`OpenSteamTool-v1.4.3-Release.zip`）
- **下载 URL 模板**：
  ```
  https://github.com/OpenSteam001/OpenSteamTool/releases/download/<tag>/OpenSteamTool-<tag>-Release.zip
  ```
- 资产 zip 内含 `OpenSteamTool.dll` / `dwmapi.dll` / `xinput1_4.dll` 三个文件（解压后直接对应 7.1 节部署清单）

> 若上游资产命名变化，命名约定独立成 `services/upstream_dll_release.py:RELEASE_ASSET_PATTERN` 常量，方便集中调整。

#### 7.7.2 检测频率与触发

三个触发点：

1. **手动**：「安装器」页 → 「检查更新」按钮（必有）
2. **启动时**：进入主窗口后异步检查一次（设置可关闭，默认开）
3. **定时**：每 24h 后台检查一次（设置可调：从不 / 启动时 / 每日，默认每日）

不做：每次启动都强制阻塞检查 / 启动期对话框 / 强制升级。检查失败完全静默（写 `network.log`），不打断用户。

#### 7.7.3 API 链与镜像

主接口：

```
GET https://api.github.com/repos/OpenSteam001/OpenSteamTool/releases/latest
```

返回 JSON 提取 `tag_name`、`name`、`body`（changelog）、`assets[].browser_download_url`。

**镜像 fallback**（参照 9.5 节，CN 检测命中后优先）：
- `https://gh-proxy.org/https://api.github.com/repos/.../releases/latest`
- `https://cdn.gh-proxy.org/...`
- `https://edgeone.gh-proxy.org/...`
- `https://ghp.ci/...`
- 直连 fallback：`api.github.com`

GitHub API 请求带可选 `Authorization: Bearer <Github_Personal_Token>`（设置项），无 token 时 60 req/h（够用，因为我们最多 24h 一次）。

#### 7.7.4 版本比较

- 解析 `tag_name`（去掉 `v` 前缀）→ 与 `dlls/manifest.json` 中 `version` 字段比较（SemVer）
- 仅在「上游 > 本地」时提示更新，不做降级提示
- 若用户当前 DLL 是用户手填的本地文件（无 manifest.json），跳过版本比较，仅在 UI 上以「上游最新版：vX.Y.Z」展示信息

#### 7.7.5 UI 表现

- 状态栏右侧出现「⬆ vX.Y.Z 可用」chip（点击跳安装器页）
- 安装器页顶部新增「内置版本 vX.Y.Z · 上游最新 vA.B.C · [更新到 vA.B.C]」横幅
- 设置 → 关于 → 「检查更新」按钮 + 上次检查时间

不弹模态对话框打断用户（与 11.5 错误处理哲学一致）。

#### 7.7.6 下载与替换流程

用户点「更新到 vA.B.C」后：

1. **预检**：与 7.4 安装前安全检查同（Steam 进程 / 路径 / 管理员）
2. **下载**：
   - URL 按 7.7.1 模板拼出
   - CN 时走镜像 fallback：`gh-proxy.org/https://github.com/...` 等（与 9.5 同一镜像列表）
   - 下载到临时目录 `<gui_data>/cache/upstream_dlls/<tag>.zip`
   - 显示进度条（彩箔扫光样式，对接 6.7.5 进度组件）
   - 支持中途取消
3. **校验**：
   - 优先：若 GitHub Release 提供 `*.sha256` 资产或在 `body` 中嵌入哈希则比对
   - 兜底：检查 zip 完整性（`zipfile.testzip()`）+ 三个 DLL 都存在 + 大小合理（>10KB 各文件）
   - 不做 GPG 签名校验（上游目前未签名）
4. **解压到 staging 区**：`<gui_data>/cache/upstream_dlls/<tag>/`（不直接覆盖 Steam 目录）
5. **接管为「在线更新源」**：把 staging 区作为新的优先级 1 来源，覆盖 7.2 节的「内置」
   - 写入 `<gui_data>/active_dll_source.json`：`{ "source": "online", "version": "1.4.3", "path": "...", "downloaded_at": "..." }`
   - 之后 GUI 重启或重新打开安装器页，状态显示「来源：在线更新 v1.4.3」
6. **触发安装**：复用 7.4 / 7.5 的安装流程（Steam 进程检测 / 备份原文件 / 拷贝），把 staging 区的 DLL 写入 `<Steam>/`

任何一步失败 → 回滚（删 staging 区 + active_dll_source.json 还原），错误信息显示在安装器页（不弹对话框）。

#### 7.7.7 切换回内置 / 本地

安装器页提供「DLL 来源」三选一切换：

- 内置（v<gui_built_version>）
- 在线更新（v<latest_downloaded>，若已下载）
- 本地文件（用户自选路径）

切换仅改变下次安装时的来源，不会立刻覆盖已写入 Steam 目录的文件。状态显示当前激活源 + 已写入 Steam 目录的源（两者可能不同，UI 显式标注「Steam 目录中是 v1.4.2，下次安装将写入 v1.4.3」）。

#### 7.7.8 缓存与清理

- 已下载的上游 DLL 保留最近 3 个版本（按下载时间排序，超出删最旧）
- 设置 → 缓存：「上游 DLL：N.N MB」+ 一键清空
- 缓存清空只删 `upstream_dlls/`，不影响已写入 Steam 目录的文件

#### 7.7.9 错误处理矩阵

| 失败点 | UI 表现 | 内部处理 |
|--------|---------|----------|
| API 请求所有镜像都失败 | 状态栏不显示 chip | 写 network.log，下次定时再试 |
| Release 资产命名不匹配 | 安装器页 banner：「检测到新版本但下载链接异常，请手动下载」+ 跳转按钮（指向 GitHub Release 页面） | 退化为引导用户去 Release 页 |
| 下载中断 | 进度区显示「下载失败，重试」按钮 | 删除半成品文件 |
| 校验失败 | 「下载文件校验失败，已自动删除」 | 删除 zip + 临时解压目录 |
| 解压失败 | 同上 | 同上 |
| 写入 Steam 失败（权限/锁） | 复用 7.4 的处理（提示关闭 Steam / 提权） | 保留 staging 区供下次重试 |

#### 7.7.10 与 GUI 自身更新的区分

7.7 节仅处理 **DLL** 的在线更新。GUI 自身（PySide6 应用本体）的更新仍按 14.6：

- GUI 顶部「关于」对话框可显示「GUI 有新版本可用：vX.Y.Z」链接 → 跳转 GitHub Release 页
- GUI 不自动下载替换自身（涉及替换正在运行的 exe，工程量与签名风险过高，留 v2）

> 简言之：DLL 在线更新在 v1 做（路径短、收益大、风险低），GUI 自更新不做（路径长、收益边际）。

## 10. D 模块（二）：添加游戏工作流

### 10.1 三入口共用 pipeline

| 入口 | 触发 |
|------|------|
| A 单加 | 「+ 添加游戏」按钮 → 输 appid 或粘 SteamDB / Steam URL |
| B 批量 | 「批量导入」按钮 → 文本框粘多个 appid（一行一个） |
| C 搜索 | 顶部搜索框输游戏名 → CaiGames `/search` → 候选列表 → 点选 |

### 10.2 四步 pipeline

```
[1 输入]
  appid + 选项（自动包含 DLC / 获取 token / depot key / manifest GID）
    ↓
[2 拉元信息]（自动）
  Steam appdetails → CaiGames → SteamCMD（任一成功即继续）
    ↓
[3 结果调整]（仅 A 显示）
  DLC 复选框列表 + Depot key 命中状态（✓ 找到来源 / ⚠ 未找到需手填）
  用户可取消勾选不想要的 DLC
    ↓
[4 预览 + 完成]
  生成的 .lua 内容（只读预览） → 点「添加到库」
```

### 10.3 核心约束

- 所有步骤允许中途取消（已拉部分存 DB，标记 incomplete）
- 失败不阻塞：token 拉不到 → 警告 + 保留其他配置
- **缺 key 的 depot 显式标记**：主表「⚠ 部分配置缺失」chip
- 用户取消勾选的 DLC **一定不入库**（不自动猜测兜底）
- B 模式后台并发限流 5；完成后报告（成功 / 部分缺失 / 失败）→「全部添加」批量入库

## 11. D 模块（三）：网络层与缓存

### 11.1 HTTP 客户端

- `httpx.AsyncClient` 全局单例
- `verify=False` 默认（cysaw / sudama 等镜像证书可能不严，参照 `Fluent-Install/backend/cai_backend.py:156`）
- 设置项「严格 SSL」开关给安全敏感用户
- `trust_env=True`：自动用系统 `HTTP_PROXY` / `HTTPS_PROXY`
- 设置项「自定义代理」（覆盖系统代理）

### 11.2 速率限制（按数据源）

| 源 | 限速 |
|----|------|
| Steam appdetails | 1 req/s（IP 级 200/5min） |
| GitHub API（有 token） | 5000/h（不主动限流，遇到 429 退避） |
| GitHub API（无 token） | 禁用 GitHub 路径（提示用户配 token） |
| CaiGames / SteamCMD / Sudama | 不限流，依赖批量 + 缓存 |
| 全局并发 | 5 |

### 11.3 重试

每源 3 次指数退避（1s / 2s / 4s），失败后切下个源。

### 11.4 三层缓存

| 层 | 内容 | TTL |
|----|------|-----|
| 内存 LRU 512 entries | 启动期跨源去重 | 进程生命周期 |
| `<gui_data>/cache/sudama_keys.json` | 全量 depot keys | 24h |
| `<gui_data>/cache/sudama_tokens.json` | 全量 access tokens | 24h |
| `<gui_data>/cache/appdetails/{appid}.json` | Steam 元信息 | 7d |
| `<gui_data>/cache/headers/{appid}.jpg` | 封面图 | 永久 |

设置 → 缓存：查看大小 + 按类型清空。

### 11.5 错误处理层级

- 单源失败 → 静默切下个源
- 所有源失败 → UI「无法获取 appid X」+ 详细日志（不弹对话框打断）
- 字段缺失 → 入库 + incomplete 标记 + 主表「⚠」chip
- **不做**：自动 fallback 到 SteamDB scraping、错误对话框打断、主表红色错误图标

### 11.6 网络日志

- `<gui_data>/logs/network.log`
- token / key 脱敏（前 4 后 4 字符）
- 默认 INFO 级别，设置可调 DEBUG

## 12. 安全与原子性

### 12.1 Lua 文件原子写

```python
def atomic_write(path: Path, content: str) -> None:
    tmp = path.with_suffix(path.suffix + '.tmp')
    tmp.write_text(content, encoding='utf-8', newline='\n')
    os.replace(tmp, path)  # MoveFileEx + REPLACE_EXISTING
```

启动时扫描 `managed/` 删除残留 `.tmp`（崩溃恢复）。

### 12.2 SQLite 并发

- WAL 模式：`PRAGMA journal_mode=WAL`
- 写操作走事务（DB 记录 + 文件操作打包，失败回滚）
- 单进程内 `QSqlDatabase` 连接池

### 12.3 Steam 运行时写盘

**允许写入，不阻止**。理由：DLL 设计哲学就是 hot reload（FileWatcher + 引用计数 + 增量 unload/reload，见 `LuaConfig.cpp:543-571`）。

GUI 仅保证写盘原子，状态栏显示「Steam 正在运行 · 修改立即生效」让用户知情。

### 12.4 注册表写入

- `RegSetValueEx` 单值原子，无需特殊处理
- 写前备份当前值到 `registry_backups` 表
- 写前校验 hex 长度 + 可解码，不合法不写
- 支持「恢复上次注册表值」按钮

### 12.5 路径与编码

- 全部用 `pathlib.Path`
- Lua 文件：UTF-8 BOM-less + LF 换行
- 所有文本文件统一 UTF-8

### 14.4 CI（GitHub Actions）

`.github/workflows/gui-release.yml` 大致结构：

```yaml
name: Build GUI Release
on:
  push:
    tags: ['v*']
  pull_request:
    paths: ['gui/**', 'src/**']
jobs:
  build:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build C++ DLL (Release)
        run: ./build.bat
      - name: Stage DLL into GUI resources
        run: |
          mkdir -p gui/src/opensteamtool_gui/resources/dlls
          cp build/Release/*.dll gui/src/opensteamtool_gui/resources/dlls/
          # 生成 manifest.json（哈希 + 版本）
      - uses: astral-sh/setup-uv@v3
      - name: Setup Python
        uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - name: Install deps & compile i18n
        working-directory: gui
        run: |
          uv sync
          uv run pyside6-lrelease i18n/*.ts
      - name: Build exe
        working-directory: gui
        run: uv run python build_exe.py
      - name: Zip
        run: 7z a OpenSteamTool-GUI-${{ github.ref_name }}.zip ./gui/dist/OpenSteamTool-GUI/
      - uses: actions/upload-artifact@v4
        with:
          name: OpenSteamTool-GUI
          path: '*.zip'
      - name: Create Release
        if: startsWith(github.ref, 'refs/tags/v')
        uses: softprops/action-gh-release@v2
        with:
          files: '*.zip'
```

### 14.5 版本号

- GUI 与 DLL **共用 SemVer**，一个 tag 同时发版
- 三处保持一致：
  - `gui/pyproject.toml` → `[project] version`
  - `src/CMakeLists.txt:2` → `project(OpenSteamTool VERSION ...)`
  - `gui/src/opensteamtool_gui/resources/dlls/manifest.json` → CI 生成

### 14.6 v1 不做清单

- **GUI 自身**自动更新（GUI 本体走「关于」对话框给链接跳 GitHub Release，用户手动下载替换）

- 代码签名（接受 SmartScreen 警告）

- 多平台（仅 Windows x64）
